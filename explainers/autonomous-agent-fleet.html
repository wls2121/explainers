How an autonomous coding-agent fleet is wired

A generic pattern, not any one company's setup. A directed graph of how a single trigger flows through an autonomous software-engineering system. Triggers converge on a durable workflow engine, which drives the agent reasoning loop, which fans out to tools, which produce outputs — and one output (a self-improvement ticket) loops back as a new trigger.

Autonomous schedules — the “default state = act” engine

These run on the engine's timers/polls with no human in the loop. A typical triage cron looks like 0 9 * * 1-5 (weekday mornings); the rest are described by cadence.

Backlog triageweekdays · 9am
Finds backlog tickets without a triaged label, runs a ticket-improvement skill, comments & labels them.
Code healthweekly / agent
Ranks tech debt, then opens refactor or test-coverage PRs nobody asked for.
Routing schedulercontinuous
Polls the tracker, picks up ready-for-agent tickets and dispatches them to the right agent.
Code reviewscheduled
Backstop that reviews open PRs in the configured repos (a per-agent review hook is the primary trigger).

Reads its own transcript after each session and files tickets on itself under a meta-epic.

Runs a nightly simulation against the platform under test to surface regressions.

The fleet — one platform, many specialized agents

The routing scheduler is the fan-out point: it reads the board & repo of each ticket and hands it to the agent that owns that domain. Each agent runs the same harness with its own repos, model config, and context file.

Read the graph in one breath: four kinds of trigger funnel into a durable workflow engine → which runs the agent reasoning loop → which calls a short list of tools → which produce outputs. The magenta loop is the part that makes it feel alive: the system reviews its own transcripts and files tickets that come back in as fresh work — no human in that cycle.
